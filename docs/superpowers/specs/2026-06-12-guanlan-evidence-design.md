# 观澜二期：候选证据可信度与规则解释设计

## 背景

观澜已经完成免费公开行情、历史、新闻和公告源接入，并将产品定位收敛为盘后复盘与信息驱动候选发现。下一步要解决的问题不是继续堆更多候选，而是让用户看懂每一只候选为什么出现、证据来自哪里、短线或中长线的判断依据是什么。

## 目标

- 每只候选都展示结构化证据链，覆盖量价图形、题材信息、新闻公告和风险排除。
- 每只候选都给出策略周期解释，说明为什么是短线、中长线或综合观察。
- 候选详情页能直接呈现证据可信度，而不是只展示分数和散落文本。
- 保持合规边界：只做复盘、观察和信息整理，不给买卖建议、目标价、仓位或收益承诺。

## 非目标

- 不接入交易账户。
- 不生成买入、卖出、止盈、止损指令。
- 不做收益预测。
- 不引入付费数据源作为默认路径。

## 后端设计

### 数据模型

在 `StealthCandidate` 上新增：

- `horizon_reason`: 策略周期解释，中文短句。
- `evidence_breakdown`: 结构化证据列表。

`evidence_breakdown` 的单项结构：

- `category`: `量价图形`、`题材信息`、`新闻公告`、`风险提示`、`数据质量`。
- `title`: 证据标题。
- `detail`: 证据说明。
- `weight`: `high`、`medium`、`low`。
- `source_ids`: 关联数据源 id 列表。

### 扫描器输出

`stealth_scanner.py` 负责从现有指标生成证据：

- 平台收敛、均线修复、平台突破、放量突破进入 `量价图形`。
- 题材归属、题材共振、信息共振进入 `题材信息`。
- 数据不足、过热、流动性不足进入 `风险提示`。
- 历史源、信息源缺失或候选证据不足进入 `数据质量`。

`horizon_reason` 的规则：

- 短线：启动分高、放量突破、强势阳线或信息共振更强。
- 中长线：潜伏分高、平台收敛、均线修复和题材共振更强。
- 综合观察：证据不足以稳定归类，继续观察。

### 持久化

先复用 `metrics` JSON 存储新增字段，避免立即扩大 SQL 表结构。Repository 读取候选时从 `metrics` 恢复字段，并对历史数据给默认值。

## 前端设计

候选详情区域新增一个“证据卡片”块：

- 顶部显示策略周期和 `horizon_reason`。
- 按证据类别分组展示 evidence cards。
- 每张卡展示标题、说明和权重。
- 如果没有结构化证据，回退显示现有 `evidence` 文本。

视觉基调保持当前 SaaS 工具风格：紧凑、可扫描、低装饰。证据卡片不要做营销式大卡，而是像研究笔记一样密集、清晰。

## API 兼容

新增字段有默认值：

- `horizon_reason = ""`
- `evidence_breakdown = []`

旧候选数据不需要迁移，读取时自动补默认值。

## 测试计划

- 后端测试：短线突破候选应产生短线解释和量价/题材证据。
- 后端测试：中长线波段候选应产生中长线解释和平台/题材证据。
- Repository 测试：从旧 metrics 读取候选时新增字段保持默认值。
- 前端类型检查：`npm run build` 覆盖新增字段和展示逻辑。
- 全量验证：`python -m pytest backend/tests -q`、`python -m compileall -q backend tools`、`npm run build`。

## 验收标准

- 候选详情能看到“为什么是短线或中长线”。
- 候选详情能看到至少一组结构化证据卡片。
- 没有结构化证据的旧数据仍能正常渲染。
- CI 继续通过。
- PR 模板里的数据源和合规边界不被破坏。
